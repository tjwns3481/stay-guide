## 1. 문제 정의

### 1.1 현재 상황 (As-Is)

**호스트의 고통:**
감성 숙소를 운영하는 30대 호스트 민지는 매일 같은 질문에 시달립니다. "와이파이 비밀번호가 뭐예요?", "주차는 어디에 하나요?", "근처 맛집 추천해주세요." 심야에도 카톡이 울리고, 새벽에 보일러 작동법을 묻는 손님 때문에 잠을 설칩니다.

종이 안내문을 만들어봤지만 손님들이 잘 안 보고, 와이파이 비밀번호가 바뀌면 다시 인쇄해야 합니다. 노션으로 디지털화해봤지만, 모바일에서 보면 딱딱하고 예쁘지 않아서 숙소의 감성이 깨집니다.

**손님의 불편:**
손님 준호는 1박 숙박을 위해 전용 앱을 설치하거나 회원가입하는 게 귀찮습니다. 그냥 링크 하나로 모든 정보를 바로 보고 싶을 뿐입니다.

### 1.2 목표 상황 (To-Be)

**호스트 민지의 변화:**
- 예약 확정 시 `roomy.com/minjis-house` 링크 하나만 보내면 끝
- 손님이 알아서 정보를 찾아보고, 반복 문의가 90% 감소
- 와이파이 비밀번호가 바뀌면 관리자 페이지에서 수정하면 즉시 반영
- 모바일 청첩장처럼 예쁜 가이드북으로 숙소 브랜딩까지 강화

**손님 준호의 변화:**
- 앱 설치 없이 링크 클릭 한 번으로 모든 정보 확인
- 와이파이 비밀번호 복사 버튼으로 바로 연결
- 주차장/맛집 위치는 지도에서 바로 확인

### 1.3 왜 지금인가? (Why Now)

- 에어비앤비/감성 숙소 시장이 지속 성장 중
- 기존 솔루션(노션, 종이 안내문)의 한계가 명확함
- AI 기술 발전으로 향후 자동 응답 기능 확장 가능
- 사업적 기회 발견: 모바일 청첩장처럼 숙박업소용 SaaS 시장이 비어있음

## 2. 사용자 페르소나

### 2.1 주요 페르소나 (호스트)

| 항목 | 내용 |
|------|------|
| 이름/별칭 | "감성 숙소 운영자 민지" |
| 나이/직업 | 32세, 에어비앤비 호스트 (부업 또는 전업) |
| 핵심 니즈 | 반복 문의 없이 손님이 알아서 정보를 찾아보길 원함 |
| 불편 포인트 | 심야 문의, 같은 질문 반복, 종이 안내문 수정 번거로움 |
| 사용 상황 | 예약 확정 후 링크 전송, 정보 수정 시, 새 숙소 추가 시 |

### 2.2 최종 사용자 (손님)

| 항목 | 내용 |
|------|------|
| 이름/별칭 | "편한 게 좋은 손님 준호" |
| 나이/직업 | 28세, 직장인 |
| 핵심 니즈 | 앱 설치 없이 링크로 바로 정보 확인 |
| 불편 포인트 | 1박 숙박인데 앱 설치/회원가입 요구, 정보 찾기 어려움 |
| 사용 상황 | 체크인 전 정보 확인, 체류 중 와이파이/맛집 검색 |

### 2.3 페르소나의 하루 (시나리오)

**오후 3시 - 예약 확정**
민지는 새 예약이 들어왔다는 알림을 받습니다. 관리자 페이지에서 숙소의 가이드북 링크를 복사해 예약 확정 문자와 함께 보냅니다. "안녕하세요! 예약해주셔서 감사합니다. 숙소 안내는 이 링크를 확인해주세요: roomy.com/minjis-house"

**오후 5시 - 손님의 체크인**
손님 준호는 숙소에 도착해서 링크를 클릭합니다. 예쁜 디자인의 가이드북이 열리고, 제일 위에 있는 와이파이 블록에서 "복사" 버튼을 눌러 바로 연결합니다. 주차장 위치도 지도에서 바로 확인합니다.

**밤 11시 - 평화로운 저녁**
예전 같으면 "와이파이 비번이요?", "보일러 어떻게 켜요?" 문의가 왔겠지만, 손님이 가이드북에서 알아서 찾아봐서 민지의 저녁은 평화롭습니다.

## 3. 사용자 스토리

### FEAT-0: 공통 흐름 (온보딩/인증)
- [ ] 호스트로서, 이메일로 간편하게 가입할 수 있어야 한다. 왜냐하면 빠르게 서비스를 시작하고 싶기 때문이다.
- [ ] 호스트로서, 로그인 상태가 유지되어야 한다. 왜냐하면 매번 로그인하는 게 번거롭기 때문이다.
- [ ] 손님으로서, 로그인 없이 가이드북을 볼 수 있어야 한다. 왜냐하면 1박 숙박에 회원가입은 부담스럽기 때문이다.

### FEAT-1: 필수 정보 블록 (MVP 핵심)
- [ ] 호스트로서, 와이파이 정보(SSID, 비밀번호)를 입력할 수 있어야 한다. 왜냐하면 가장 많이 받는 질문이기 때문이다.
- [ ] 손님으로서, 와이파이 비밀번호를 한 번 탭으로 복사할 수 있어야 한다. 왜냐하면 긴 비밀번호를 직접 입력하기 귀찮기 때문이다.
- [ ] 호스트로서, 주차장/숙소 위치를 지도로 표시할 수 있어야 한다. 왜냐하면 "어디에 주차해요?" 문의를 줄이고 싶기 때문이다.
- [ ] 손님으로서, 지도에서 위치를 확인하고 내비게이션으로 바로 연결할 수 있어야 한다. 왜냐하면 편하게 찾아가고 싶기 때문이다.
- [ ] 호스트로서, 체크인/체크아웃 시간을 명시할 수 있어야 한다. 왜냐하면 기본적인 규칙 안내가 필요하기 때문이다.
- [ ] 호스트로서, 주변 맛집/카페/관광지를 추천할 수 있어야 한다. 왜냐하면 손님 경험을 높이고 문의를 줄이고 싶기 때문이다.

### FEAT-2: 모바일 가이드북 (MVP)
- [ ] 호스트로서, 감성적인 템플릿을 선택할 수 있어야 한다. 왜냐하면 숙소의 분위기와 어울리는 디자인이 필요하기 때문이다.
- [ ] 호스트로서, 숙소 사진을 업로드할 수 있어야 한다. 왜냐하면 가이드북에 숙소 분위기를 담고 싶기 때문이다.
- [ ] 손님으로서, 모바일에서 빠르게 로딩되는 가이드북을 볼 수 있어야 한다. 왜냐하면 느린 페이지는 답답하기 때문이다.
- [ ] 손님으로서, 앱 설치 없이 링크로 바로 접속할 수 있어야 한다. 왜냐하면 1박에 앱 설치는 부담스럽기 때문이다.

### FEAT-3: 관리자 페이지 (MVP)
- [ ] 호스트로서, 가이드북 정보를 언제든 수정할 수 있어야 한다. 왜냐하면 와이파이 비밀번호나 맛집 정보가 바뀔 수 있기 때문이다.
- [ ] 호스트로서, 수정 내용이 즉시 반영되어야 한다. 왜냐하면 손님이 항상 최신 정보를 봐야 하기 때문이다.
- [ ] 호스트로서, PC와 모바일 모두에서 관리할 수 있어야 한다. 왜냐하면 이동 중에도 수정이 필요할 수 있기 때문이다.

### FEAT-4: AI 컨시어지 챗봇 (v2 이후)
- [ ] 손님으로서, 가이드북에 없는 정보는 AI에게 물어볼 수 있어야 한다. 왜냐하면 호스트에게 직접 연락하기 전에 빠르게 답을 얻고 싶기 때문이다.
- [ ] 호스트로서, 심야에도 AI가 기본 문의를 자동 응대해야 한다. 왜냐하면 잠을 방해받고 싶지 않기 때문이다.

## 5. 범위 관리

### 5.1 MVP (반드시 포함)
- **FEAT-1**: 필수 정보 블록 (와이파이, 지도, 체크인/아웃, 맛집)
- **FEAT-2**: 모바일 가이드북 (감성 UI, SSR 최적화)
- **FEAT-3**: 관리자 페이지 (정보 입력/수정, 반응형)
- **FEAT-0**: 기본 온보딩 (이메일 가입)

### 5.2 옵션 (시간 허용 시)
- 다중 숙소 관리 (한 계정에 여러 숙소)
- 템플릿 추가 선택지
- 방문자 통계 대시보드

### 5.3 Non-goals (명시적 제외)
- **AI 챗봇**: v2에서 프리미엄 기능으로 추가. MVP에서는 정보 제공에 집중.
- **결제 연동**: 초기에는 외부 결제(스마트스토어 등)로 처리. PG 연동은 v2.
- **소셜 로그인**: MVP에서는 이메일 가입만. 간소화 우선.
- **다국어 지원**: MVP에서는 한국어만. 해외 확장은 v2.

## 7. 해결 가설과 실험

### 7.1 가설
> 만약 **모바일 청첩장 스타일의 예쁜 가이드북**을 제공하면, **노션을 쓰던 호스트가 전환**할 것이다. 왜냐하면 **숙소의 감성을 해치지 않는 UI가 핵심 니즈**이기 때문이다.

### 7.2 실험 루프

```
실험 → 관측 → 학습 → 다음 가설
```

| 단계 | 내용 |
|------|------|
| 실험 | 랜딩 페이지 + 데모 가이드북으로 베타 테스터 모집 |
| 관측 | 가입 전환율, 가이드북 완성율, 피드백 수집 |
| 학습 | 전환율 10% 이상이면 성공, 미만이면 UI/온보딩 개선 |
| 다음 | "AI 챗봇이 추가되면 프리미엄 전환율이 올라갈까?" |
